Persistent Message Queues With Replay
  • REST API for easy access from curl and any programming language
  • Replay messages by timestamp or id
  • Use on its own or as a front end to RabbitMQ

Backup/Replay RabbitMQ Queue

  • Backup messages from a queue
  • Replay by timestamp as needed
  • Watch incoming messages with curl

Push To Remote RabbitMQ Server

  • Simple HTTP POST to send messages to RabbitMQ
  • Your app is protected from network partitions or RabbitMQ downtime
  • Replay by timestamp as needed if downstream processing fails
  • Easier than running RabbitMQ on both servers with shovels
  • Low memory footprint (run on an EC2 micro instance)

The following example creates a queue and configures it to output to RabbitMQ:

{% highlight bash %} $ curl -XPOST http://127.0.0.1:9554/q/incoming $ curl -XPOST http://127.0.0.1:9554/q/incoming/out/rabbit \ -d type=rabbitmq -d url=amqp://127.0.0.1 -d queues=incoming # these operations are idempotent i.e. if the object already exists it is updated {% endhighlight %}

Its easy to POST messages to the queue:

{% highlight bash %} $ curl -XPOST http://127.0.0.1:9554/q/incoming/messages?routingKey=abc -d @message.json # the contents of message.json are now in RabbitMQ {% endhighlight %}

And stream them out again or redeliver:

{% highlight bash %} $ curl http://127.0.0.1:9554/q/incoming/messages?from=10:13 # messages since 10:13 today are streamed to the console $ curl -XPOST http://127.0.0.1:9554/q/incoming/out/rabbit -d from=2013-06-24 # messages since 24 June 2013 are redelivered to RabbitMQ {% endhighlight %}
